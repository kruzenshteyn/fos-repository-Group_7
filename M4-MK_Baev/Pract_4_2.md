## 3. Часть 1. Развертывание модели на ESP32‑S3 (40 минут)

### 3.1. Выбор задачи: Wake Word Detection или Gesture Recognition

Студенты делятся на две группы:

**Группа А** — распознавание ключевого слова (wake word)
- Микрофон → MFCC-признаки → компактная CNN
- Модель: 2 класса ("wake word" / "background")
- Размер модели после int8‑квантизации: **100–300 КБ**

**Группа Б** — классификация жестов
- Акселерометр MPU6050 → 6-осевой временной ряд → Conv1D
- Модель: 3–5 классов жестов (OK, Fist, Open, Up/Down sweep)
- Размер модели: **~110 тыс. параметров, 437 КБ (float32)**

### 3.2. Подготовка модели (Python)

**Шаг 1. Обучение модели на ПК**

```python
# train_model.py — пример для распознавания ключевых слов
import tensorflow as tf
import numpy as np

# Компактная CNN для MFCC-признаков (20×10)
model = tf.keras.Sequential([
    tf.keras.layers.Conv2D(16, (3,3), activation='relu', input_shape=(20,10,1)),
    tf.keras.layers.DepthwiseConv2D((3,3), activation='relu'),
    tf.keras.layers.Flatten(),
    tf.keras.layers.Dense(32, activation='relu'),
    tf.keras.layers.Dense(2, activation='softmax')  # 2 класса
])

model.compile(optimizer='adam', loss='sparse_categorical_crossentropy', metrics=['accuracy'])
model.fit(x_train, y_train, epochs=20, validation_data=(x_val, y_val))
Шаг 2. Квантизация и конвертация в TFLite

Для микроконтроллеров без FPU критически важно использовать полноцелочисленную квантизацию (int8):

python
# convert_model.py
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.target_spec.supported_types = [tf.int8]
converter.representative_dataset = representative_dataset_gen  # калибровочные данные
tflite_quant_model = converter.convert()

with open('model.tflite', 'wb') as f:
    f.write(tflite_quant_model)
Шаг 3. Преобразование в C‑массив

bash
xxd -i model.tflite > model_data.cc
Результат: модель размером 240 КБ (вместо 1.2 МБ)

3.3. Интеграция в прошивку ESP32
Для Arduino IDE (рекомендуемый способ для быстрого старта) :

cpp
// tinyml_wake_word.ino
#include <TensorFlowLite_ESP32.h>
#include "model_data.h"  // Сгенерированный C-массив

const tflite::Model* model = tflite::GetModel(model_data);
tflite::MicroInterpreter interpreter(model, resolver, tensor_arena, kTensorArenaSize);

void setup() {
    Serial.begin(115200);
    interpreter.AllocateTensors();
    Serial.println("TinyML Wake Word Detector ready!");
}

void loop() {
    // 1. Захват аудио через I2S (16 кГц, 40 мс фреймы)
    // 2. Вычисление MFCC (2–3 мс на ESP32-S3)
    // 3. Заполнение входного тензора
    // 4. Инференс
    interpreter.Invoke();
    
    // 5. Чтение результата
    uint8_t score = output->data.uint8[1];  // уверенность для "wake word"
    if (score > 200) {
        Serial.println("Wake word detected!");
    }
}
Для ESP‑IDF (профессиональный подход) :

Проект включает компоненты: main/ с задачами FreeRTOS для сбора данных с датчика, инференса и передачи результата по BLE.

3.4. Настройка ESP‑NN для максимальной производительности
ESP‑NN — это библиотека оптимизированных ядер для нейросетевых операций на чипах Espressif. Она включена по умолчанию через флаг -DESP_NN.

В Arduino IDE: библиотека TensorFlowLite_ESP32 автоматически использует ESP‑NN.

В ESP‑IDF: в menuconfig → ESP-NN → выбрать NN_OPTIMIZATIONS.

Эффект ESP‑NN на производительность:

Целевая платформа	Без ESP‑NN	С ESP‑NN	Ускорение
ESP32‑S3	2300 мс	54 мс	42.6×
ESP32	4084 мс	380 мс	10.7×
ESP32‑P4	1395 мс	73 мс	19.1×
Важно: ESP32‑S3 имеет векторный набор инструкций (PIE), ускоряющий операции умножения-накопления.

4.3. Измерение энергопотребления
Схема измерения:

text
[Источник питания 3.3В] ---- [Резистор 9.8 Ом] ---- [ESP32]
                                   |
                                  Вольтметр
Ток вычисляется по закону Ома: I = V / R

Ожидаемые значения:

Режим	Потребление
Пиковая активная мощность (инференс)	~280 мВт
Среднее (с duty‑cycling)	< 10 мВт
Режим глубокого сна	~0.01 мВт
Для справки: потребление ESP32 в режиме инференса составляет 0.269–0.327 Вт.

5.2. OTA-обновления — ключевая инженерная практика
В продуктивной среде модели должны обновляться без физического доступа к устройству:

Новая модель обучается и оптимизируется в облаке

Обновление доставляется через OTA (Over‑The‑Air)

Устройство продолжает работу с обновлённой моделью

Это критически важно для компетенции LC‑5.1 (средний уровень): специалист должен не просто запустить модель, а настроить процессы её обновления и мониторинга.

6.3. Когда выбирать Raspberry Pi Zero?
Требуется больше памяти для модели (> 2 МБ)

Нужна поддержка Linux и существующих библиотек

Модель не умещается в ограничения TFLM

Энергопотребление не критично
